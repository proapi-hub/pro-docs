## 章节导读

本页是 Codex CLI 接入 **ProAPI**（`newapi.prorisehub.com`）时社区反馈最高频的问题合集。涵盖 **配置不生效类高频坑**、**网络与 MCP 启动**、**上下文与成本**、**安装与登录** 四大方向。

**基础配置请先阅读**：[Codex 配置](/cli/codex) 或 [CC-Switch 接入](/ccswitch/codex)。

<Callout type="info" title="令牌分组优先">
  Codex CLI 必须使用 ProAPI 的 **Codex 专用分组** 令牌，分组选错是「401 / usage_not_included / model not found」类问题的最常见根因。详见 [令牌与分组](/token/intro)。
</Callout>

## 高效使用 Codex

很多人在使用一段时间后认为模型「降智」。从实际使用来看，Codex 模型升级后并未变弱，关键在于 **如何合理使用模型**。

### 三条核心原则

1. **任务划分** — 任何时候都不要提交过于笼统的任务（如「请帮我写一个管理系统后台」）。Codex 的特点是 **严谨有序、指哪打哪**，意味着必须先把任务拆解到模块化粒度。

2. **掌控之内** — 提交任务前先评估它是否拆得够细，是否符合模块化开发。理想状态是：你能预估 Codex 这次会改哪些文件、产生哪些变动。一旦 AI 脱离掌控，结局多半是项目越改越乱直到重置。

3. **避免压缩** — 多数任务只用到 Codex 约 60% 的上下文就能解决。如果到 60% 还没解决甚至需要压缩，那次的任务拆分就算失败，应当回到第 1 步进一步拆细。**优秀的 Codex 选手几乎不用 `/compact`**。

<Callout type="info" title="碎碎念">
  AI 时代让很多东西变简单，但基础知识决定了 AI 的使用上限。当前阶段的 AI 只是一个非常优秀的 Copilot 角色，同一个模型在不同人手里会有截然不同的发挥。
</Callout>

## 配置指南

### Windows 下丝滑使用 Codex

<Callout type="warn" title="一并解决多个痛点">
  本方法同时解决 **读写文件、乱码、Token 消耗高、项目无记忆** 等多个问题。
</Callout>

1. 确保 Codex CLI 与 VSCode Codex 插件已正常工作。

2. 按 <kbd>Win</kbd> + <kbd>R</kbd> 打开用户目录：

   ```bash
   %userprofile%\.codex
   ```

3. 编辑 `config.toml`：

   ```toml
   model_provider = "procode"
   model = "gpt-5.4"
   model_reasoning_effort = "high"
   network_access = "enabled"
   disable_response_storage = true
   windows_wsl_setup_acknowledged = true
   model_verbosity = "high"

   [model_providers.procode]
   name = "procode"
   base_url = "https://newapi.prorisehub.com/v1"
   wire_api = "responses"
   requires_openai_auth = true
   ```

4. 编辑（或新建）`AGENTS.md`，写入全局工作指南：

   ```markdown
   # Codex 全局工作指南

   ## 回答风格
    - 回答必须使用中文
    - 对总结、Plan、Task 以及长内容输出，优先逻辑整理后用表格美观输出；普通内容正常输出
   ```

5. 启动 VSCode Codex 插件，体验新效果。

### Codex 在 Windows 系统下乱码

1. 按 <kbd>Win</kbd> + <kbd>R</kbd>，输入以下命令打开「区域」设置：

   ```bash
   intl.cpl
   ```

2. 切到「管理」选项卡，点击 **更改系统区域设置**。

3. 勾选「Beta 版：使用 Unicode UTF-8 提供全球语言支持」，确定后重启电脑。

如不希望改全局，可改用 **PowerShell 7**（`winget install --id Microsoft.PowerShell`），并在 Profile 中加：

```powershell
[Console]::InputEncoding = [System.Text.UTF8Encoding]::new()
[Console]::OutputEncoding = [System.Text.UTF8Encoding]::new()
$OutputEncoding = [System.Text.UTF8Encoding]::new()
```

### 配置全局提示词

1. 参照 [Codex 配置](/cli/codex) 前两步打开 `~/.codex` 目录。
2. 编辑 `AGENTS.md`（如不存在请手动创建），写入提示词。
3. 重启 Codex 或 VSCode 后生效。

### 启用内置网络搜索

1. 打开 `~/.codex/config.toml`。
2. 加入：

   ```toml
   [features]
   web_search_request = true
   ```

3. 重新运行 `codex` 即可使用网络搜索。

## 配置不生效类高频坑

<Accordions>
<Accordion title='config.toml 改了没生效？检查 model_provider 与 [model_providers.xxx] 是否同名'>

社区最常见的配置坑：`model_provider` 字段和 `[model_providers.xxx]` 段名 **必须完全一致**，否则 Codex 会回退到默认配置或报错。

**错误示例**：

```toml
model_provider = "proapi"       # ← 这里写 proapi

[model_providers.procode]       # ← 但段名写了 procode
base_url = "https://newapi.prorisehub.com/v1"
```

**正确示例**：

```toml
model_provider = "procode"

[model_providers.procode]
base_url = "https://newapi.prorisehub.com/v1"
wire_api = "responses"
requires_openai_auth = true
```

修改后 **必须重启终端**。可用 `/status` 验证当前生效的 provider。

</Accordion>

<Accordion title="exceeded retry limit, last status: 401 Unauthorized">

通常是 **环境变量残留覆盖了 config.toml**。先检查环境变量：

<Tabs items={['Windows', 'macOS']}>
<Tab value="Windows">

```bash
cmd /c "echo ================= OPENAI ENV CHECK ================= & ^
if defined OPENAI_API_KEY (echo OPENAI_API_KEY  = OK) else (echo OPENAI_API_KEY  = MISSING) & ^
if defined OPENAI_BASE_URL (echo OPENAI_BASE_URL = OK) else (echo OPENAI_BASE_URL = MISSING) & ^
echo ========================================================="
```

若输出非 `MISSING`，运行：

```bash
cmd /c "setx OPENAI_API_KEY \"\" & setx OPENAI_BASE_URL \"\""
```

</Tab>
<Tab value="macOS">

```bash
echo "================= OPENAI ENV CHECK ================="
[ -z "$OPENAI_API_KEY" ] && echo "OPENAI_API_KEY  = MISSING" || echo "OPENAI_API_KEY  = OK"
[ -z "$OPENAI_BASE_URL" ] && echo "OPENAI_BASE_URL = MISSING" || echo "OPENAI_BASE_URL = OK"
echo "========================================================"
```

若输出非 `MISSING`，运行：

```bash
unset OPENAI_API_KEY OPENAI_BASE_URL
```

</Tab>
</Tabs>

之后检查 `~/.codex/auth.json` 的 API Key 与 `config.toml` 的 `base_url` 是否正确：

- `auth.json` 内容为 `{"OPENAI_API_KEY": "sk-xxx"}`，**Key 前后不能有空格**。
- 中转站场景 **不推荐** 使用 `env_key` 字段，直接把 Key 放 `auth.json`。
- 在 ProAPI 后台确认令牌分组为 **Codex 专用分组**，状态为「已启用」。

</Accordion>

<Accordion title="wire_api 该选 chat 还是 responses？">

| 取值 | 适用场景 |
| --- | --- |
| `responses` | **ProAPI、官方 GPT-5.x 系列**。Codex 已弃用 chat 端点，所有 GPT-5.x Codex 模型仅在 Responses API 上可用。 |
| `chat` | 国产模型（DeepSeek、GLM、Kimi 等只实现 Chat Completions 兼容层）。 |

走错协议会报：

```text
Support for the "chat" wire API is deprecated
```

或

```text
unknown api endpoint /v1/responses
```

中转站场景默认配置：

```toml
[model_providers.procode]
base_url = "https://newapi.prorisehub.com/v1"
wire_api = "responses"
requires_openai_auth = true
```

并必须配合 `disable_response_storage = true`（见后文）。

</Accordion>

<Accordion title="model_reasoning_effort 配置后实际没生效？">

已知 bug（[GitHub Issue #17436](https://github.com/openai/codex/issues)）：CLI 启动时 `config.toml` 中的 `model_reasoning_effort` 可能 **被「上次会话最后一次手动改的值」覆盖**。

**临时方案**：

1. 启动后在 TUI 中用 `/reasoning` 手动确认一次。
2. 或使用 Profile 强制设置：

   ```toml
   [profiles.high]
   model_reasoning_effort = "high"
   ```

   启动时 `codex --profile high`。

</Accordion>

<Accordion title="approval_policy 反复弹确认框？">

配置了 `approval_policy = "never"` 但仍然不断弹审批。**五层优先级冲突**（从高到低）：

1. CLI 启动参数（`--ask-for-approval`）
2. 环境变量
3. Profile 配置
4. 项目级 `config.toml`
5. 全局 `~/.codex/config.toml`

排查顺序：

- 强制指定：`codex --ask-for-approval never --sandbox workspace-write`
- 使用 Profile：`[profiles.auto] approval_policy = "never"` + `codex --profile auto`
- VSCode 插件：在 Codex 对话框下方权限设置选「完全访问权限」
- 注意 `workspace-write` 下 `approval_policy` 已知会被静默改为 `on-request`，用 Profile 绕过

</Accordion>

<Accordion title="approval_policy / sandbox_mode 取值含义？">

**`sandbox_mode`**：

| 取值 | 含义 |
| --- | --- |
| `read-only` | 只读，不允许任何写入 |
| `workspace-write`（推荐） | 仅可写当前工作区 |
| `danger-full-access` | 全盘可写（高风险） |

**`approval_policy`**：

| 取值 | 含义 |
| --- | --- |
| `untrusted` | 每条命令都问 |
| `on-failure`（推荐） | 失败时再问 |
| `never` | 不问 |

<Callout type="warn" title="已知 bug">
  `workspace-write` 下 `approval_policy` 有时会被静默改成 `on-request`。建议改用 `[profiles.xxx]` 配置档配合 `codex --profile dev` 强制启用。
</Callout>

</Accordion>

<Accordion title="AGENTS.md 加载顺序？为什么全局规则没生效？">

加载顺序：

1. `~/.codex/AGENTS.md`（全局）
2. 仓库根 `AGENTS.md`
3. 当前 cwd 的 `AGENTS.md`

三者合并。**已知 bug**（[#8759](https://github.com/openai/codex/issues)）：CLI 有时不读全局文件，且只读仓库根那一份、不会向上递归。

**临时方案**：

- 文件名必须是 **全大写** `AGENTS.md`，不是 `Agents.md` / `agents.md`。
- 控制在 **几 KB 以内**（超过 32 KiB 会被截断）。
- 在项目根 `AGENTS.md` 用 `@~/.codex/AGENTS.md` 显式引用全局。
- 可配置 `project_doc_fallback_filenames = ["CLAUDE.md"]` 让 Codex 也读 `CLAUDE.md`。
- 用 `codex --ask-for-approval never "Summarize the current instructions."` 验证是否加载。

</Accordion>

<Accordion title="Windows 配置 MCP 时报 'newlines are unsupported in inline tables'？">

TOML 内联表格语法严格，**不能跨行**。

**错误写法**：

```toml
[mcp_servers.github]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-github"]
env = {
  "GITHUB_TOKEN" = "ghp_xxx",
  "PATH" = "..."
}
```

**正确写法**：

```toml
[mcp_servers.github]
command = "npx"
args = ["-y", "@modelcontextprotocol/server-github"]

[mcp_servers.github.env]
GITHUB_TOKEN = "ghp_xxx"
PATH = "..."
```

注意：键名是 `mcp_servers`（**下划线**），不是 `mcpServers`。

</Accordion>

<Accordion title="[profiles.xxx] 配置档怎么用？">

同一台机器在「快速改 bug」与「深度架构设计」之间切换：

```toml
experimental_use_profile = true

[profiles.quick]
model_reasoning_effort = "low"
sandbox_mode = "read-only"
approval_policy = "never"

[profiles.dev]
model_reasoning_effort = "high"
sandbox_mode = "workspace-write"
approval_policy = "on-failure"
```

启动时指定：

```bash
codex --profile dev
```

可绕过 `workspace-write` 下 `approval_policy` 被覆盖的 bug，也是规避 `model_reasoning_effort` 被会话覆盖问题的最稳方案。

</Accordion>
</Accordions>

## 网络与 MCP 启动

<Accordions>
<Accordion title="Codex 在沙盒里无法联网（npm install / git clone 全失败）">

`workspace-write` 默认 **禁止网络访问**，导致 Codex 写的脚本无法下载依赖、调 API。

**永久开启**（写入 `config.toml`）：

```toml
[sandbox_workspace_write]
network_access = true
```

**临时开启**（启动参数覆盖）：

```bash
codex -c 'sandbox_workspace_write.network_access=true'
```

日常推荐组合：`sandbox_mode = "workspace-write"` + `approval_policy = "on-request"` + `network_access = true`。

</Accordion>

<Accordion title="MCP 启动报 request timed out？">

Codex 默认 MCP 启动超时仅 **约 10 秒**，远不够 Docker MCP Toolkit、首次拉包等场景。

**调大超时**：

```toml
[mcp_servers.context7]
command = "npx"
args = ["-y", "@upstash/context7-mcp"]
startup_timeout_ms = 60000   # 60 秒
```

注意字段名是 `startup_timeout_ms`（旧版本是 `startup_timeout_sec`，单位不同）。

</Accordion>

<Accordion title="国内 npx 拉包超时导致 MCP 列表为空？">

Codex 每次启动通过 `npx` 自动拉 MCP 包，国内直连 npm 官方源超时率极高。**两个方案叠加使用**：

```bash
# 1. 切到国内 npm 镜像
npm config set registry https://registry.npmmirror.com

# 2. 全局预安装一次，让 npx 走本地缓存
npm install -g @modelcontextprotocol/server-github
npm install -g @upstash/context7-mcp
```

清 npx 缓存重试：

<Tabs items={['Windows', 'macOS / Linux']}>
<Tab value="Windows">

```powershell
Remove-Item -Recurse -Force $env:LOCALAPPDATA\npm-cache\_npx
```

</Tab>
<Tab value="macOS / Linux">

```bash
rm -rf ~/.npm/_npx
```

</Tab>
</Tabs>

</Accordion>

<Accordion title="Windows 下 MCP 服务器配置（含完整环境变量）">

Windows 下 MCP 子进程容易缺关键环境变量，导致 `program not found` 或启动失败。**完整可用模板**：

```toml
[mcp_servers.context7]
command = "cmd"
args = ["/c", "npx", "-y", "@upstash/context7-mcp"]
startup_timeout_ms = 60000

[mcp_servers.context7.env]
APPDATA = "C:\\Users\\YourName\\AppData\\Roaming"
LOCALAPPDATA = "C:\\Users\\YourName\\AppData\\Local"
HOME = "C:\\Users\\YourName"
SystemRoot = "C:\\Windows"
NODE_OPTIONS = "--dns-result-order=ipv4first"
```

`NODE_OPTIONS = "--dns-result-order=ipv4first"` 可解决 Node 在 Windows 上 IPv6 优先导致的解析超时。

</Accordion>

<Accordion title="MCP server 在 config.toml 里怎么配？">

CLI 与 VSCode 插件共用 `~/.codex/config.toml`：

```toml
[mcp_servers.context7]
command = "npx"
args = ["-y", "@upstash/context7-mcp"]
startup_timeout_ms = 30000

[mcp_servers.context7.env]
API_KEY = "xxx"
```

<Callout type="warn" title="语法严格">
  - 仅支持 STDIO transport。
  - TOML 一旦语法错（表名重复、env 用了 JSON 风格），CLI 与插件都会 **静默不加载** MCP，不会报错。
  - 改完后用 `/mcp` 命令确认列表。
</Callout>

</Accordion>

<Accordion title="Codex 在容器 / CLI 沙盒中无法联网（如无法拉安装包）？">

通常是 MTU 设置不当。终端正常但 Codex 沙盒不通时：

- 将代理（Clash 等）MTU 改为 `1500`。
- Linux 上找不到 Clash MTU 设置的用户参考：[https://linux.do/t/topic/1220328](https://linux.do/t/topic/1220328)。

</Accordion>

<Accordion title="国内服务器上没有梯子，怎么用 Codex？">

**方案 1：SSH 反向端口转发（最简单）**

本地有梯子时，把本地代理暴露给远端服务器：

```bash
# 在本地执行，把本地的 7890 代理映射到服务器的 7890 端口
ssh -R 7890:127.0.0.1:7890 user@server
```

服务器上：

```bash
export HTTPS_PROXY=http://127.0.0.1:7890
codex
```

**方案 2：直接走 ProAPI 中转站**

`base_url` 指向 `https://newapi.prorisehub.com/v1`，**国内服务器可直连**，零代理需求，这是最稳的方案。

**方案 3：SSH 远程会话（新版 Codex Desktop）**

新版 Codex 支持 `[features] remote_connections = true`，本地运行 Codex，通过 SSH 让 Codex 操作远程服务器。

</Accordion>

<Accordion title="Connection failed: error sending request for url ...">

本机网络问题，按顺序排查：

1. 本机能否访问其他网页？
2. 是否使用了网络代理（梯子）？如有请关闭。
3. 在终端直接运行 `codex` 测试，判断是不是 VSCode 插件问题，是的话重启 VSCode。
4. 若仍不行，带报错截图到群内咨询。

</Accordion>

<Accordion title="unexpected status 403 Forbidden: usage_not_included">

分组下当前路由的上游出现暂时性问题，处理方式：

1. 用 <kbd>Ctrl</kbd> + <kbd>C</kbd> 中断（VSCode 中点停止）。
2. 重新发起对话，观察是否复现（重试会自动切到其他可用线路）。
3. 重试 3 次以上无效，请带截图到群内反馈。

</Accordion>
</Accordions>

## 上下文与成本

<Accordions>
<Accordion title="GPT-5 系列 宣称 1M 上下文，为啥默认并没有这么多？">

Codex 中使用 GPT-5.5：仅支持 400K 上下文窗口（官方公告原文："In Codex, GPT‑5.5 is available for Plus, Pro, Business, Enterprise, Edu, and Go plans with a 400K context window."）

**GPT-5.5 当前即使设了 1M 也无效**，需要等官方放开
Codex CLI 默认配置 **未启用** 1M 上下文窗口，需要在 `config.toml` 显式设置：

但有一个可以绕过的 hack 方法，尽管非官方，但在社区有很多讨论


```toml
model_context_window = 1000000
model_auto_compact_token_limit = 500000
# 指向你自定义的模型目录 JSON
model_catalog_json = "/Users/你的用户名/.codex/model-catalog-500k.json"
```
**为什么 `auto_compact_token_limit` 建议设 500K 而不是 900K**？社区常常提到：超过 50% 后注意力质量明显下降，与其用满不如提前压缩。


Windows 用户路径用双反斜杠，例如：C:\\Users\\你的用户名\\.codex\\model-catalog-500k.json

运行以下命令（需要安装 jq）:
```bash
jq '(.models[] | select(.slug=="gpt-5.5") | .context_window)=500000' \
~/.codex/models_cache.json > ~/.codex/model-catalog-500k.json
```



</Accordion>


<Accordion title="model_reasoning_effort 五档（minimal / low / medium / high / xhigh）怎么选？">

| 取值 | 适用场景 |
| --- | --- |
| `minimal` / `low` | 补全、改注释、跑脚本 |
| `medium`（默认） | 常规改 bug、小型重构 |
| `high` | 架构设计、复杂排错 |
| `xhigh` | 仅 `gpt-5.1-codex-max` / `gpt-5.2` 支持，其他模型会报 `unknown variant xhigh` |

在 `~/.codex/config.toml` 设 `model_reasoning_effort = "high"`，会话内可用 `/reasoning` 临时切换。

</Accordion>

<Accordion title="GPT-5.x 系列模型选择困惑">

主流选择：

| 模型 | 适用场景 |
| --- | --- |
| **GPT-5.5** | 当前 frontier 模型，复杂推理与最高质量任务 |
| **GPT-5.4** | 日常编码主力，性价比最高 |
| **GPT-5.4-mini** | 子代理 / 轻量任务，便宜快 |
| **GPT-5.3-codex（已下线）** | 长时间无人值守的 agent loop，专为 agentic 调优 |

`model_reasoning_effort` 配合：日常 `medium`，简单任务 `low`，复杂重构 `high`。

可用模型名以 ProAPI [模型广场](/token/intro) 为准。

</Accordion>

<Accordion title="model_verbosity 与 token 消耗的关系？">

`model_verbosity = "low" | "medium" | "high"` 控制 GPT-5 Responses API 输出冗余度，**仅 GPT-5 系列生效**。

GPT-5.5 上 token 暴涨多见，降到：

```toml
model_verbosity = "low"
model_reasoning_effort = "medium"
```

可显著减少消耗。

</Accordion>

<Accordion title="fast_mode 加速但代价是 2 倍 token，怎么权衡？">

`fast_mode = true` 让模型响应提速 1.5x，但消耗 **是标准模式的 2-2.5x**。20 分钟就能跑完一天额度。

**不建议盲目开启**。优化顺序：

1. **压缩上下文**（`/compact`、`/clear`）
2. **关闭无关 MCP**（`/mcp` 启停）
3. **换小模型**（GPT-5.4-mini）
4. **最后才开 `fast_mode`**

使用 API Key 时走标准 API pricing，**不能使用 fast_mode credits**。

</Accordion>

<Accordion title="disable_response_storage = true 到底起什么作用？">

默认 OpenAI Responses API 会在服务端保存对话，以支撑 ZDR / 思维链复用。**中转厂商无此存储能力**，开启后 Codex 不再依赖 `previous_response_id`，改为每轮发送完整上下文。

**不加这条，中转必报 `response not found`**。所有第三方中转场景都必须设：

```toml
disable_response_storage = true
```

</Accordion>

<Accordion title="长会话 + high reasoning_effort 出现 524 / 首字慢 / 无限重试？">

经典故障链：

1. 会话上下文堆积到很大（`/status` 显示接近满）
2. `model_reasoning_effort = "high"` 让 GPT-5.x 在 Responses API 端长时间 reasoning
3. 模型 prefill + reasoning 阶段长达 2-5 分钟，**期间没有任何字节经过 CDN**
4. 触发 `Cloudflare Error 524 / origin_response_timeout`
5. Codex CLI 把 524 当 retryable 错误，反复重试，每次都撞墙

**根因不在你这边**：Cloudflare 边缘默认 100 秒 `proxy_read_timeout`（Free/Pro/Business 计划均为硬限制，不可调）。Responses API 即使已经返回 200，只要 100 秒内没有字节流出，CDN 就会切断。

**用户侧能立刻做的**：

| 优先级 | 操作 | 效果 |
| --- | --- | --- |
| ★★★ | 出现首字慢/重试时立刻 <kbd>Ctrl</kbd> + <kbd>C</kbd> 中断 | 避免重试雪崩 |
| ★★★ | `/compact` 或直接 `/new` 重开会话 | 直接消除长 prefill 根源 |
| ★★★ | 临时 `/reasoning low` 或 `/reasoning medium` | reasoning 阶段显著缩短 |
| ★★ | 复杂任务用 GPT-5.4 而非 GPT-5.5 | 5.5 reasoning 通常更深更慢 |
| ★★ | 单次任务遵循「优秀 Codex 选手避免压缩」原则——任务拆细 | 从源头避免大上下文 |
| ★ | `model_verbosity = "low"` 减少输出冗余度 | 输出阶段也变快 |

**已经卡住怎么办**：

1. <kbd>Ctrl</kbd> + <kbd>C</kbd>（或 VSCode 中点停止）强制中断。
2. `/new` 开新会话，把关键决策手动粘贴或写入 `AGENTS.md` 后重启。
3. 若反复出现，反馈到 ProAPI 群内，附 `ray_id` 与 `instance` 字段（524 错误体中有），便于服务端定位。

<Callout type="info" title="服务端侧也在持续优化">
  我们已知该问题与 Cloudflare 边缘 100 秒 read timeout 相关，正在通过 SSE keep-alive 心跳改造彻底解决。用户侧上述策略是「即时止血」，不依赖服务端修复也能跑通。
</Callout>

</Accordion>
</Accordions>

## 安装与登录

<Accordions>
<Accordion title="npm install -g @openai/codex 报 EACCES 权限错误">

**不要用 `sudo npm install`**！会污染全局 Node 权限，后续不停遇到权限问题。

**推荐方案 1：用 nvm 管理 Node**

```bash
# 安装 nvm 后
nvm install 20
nvm use 20
npm install -g @openai/codex
```

**推荐方案 2：重定向 npm 全局目录**

```bash
npm config set prefix ~/.local
echo 'export PATH=~/.local/bin:$PATH' >> ~/.zshrc
source ~/.zshrc
npm install -g @openai/codex
```

</Accordion>

<Accordion title='codex: command not found（安装完找不到命令）'>

npm 全局安装路径未加入 `PATH`。

```bash
# 查看 npm 全局 bin 目录
npm config get prefix

# 假设输出 /Users/you/.npm-global
echo 'export PATH=/Users/you/.npm-global/bin:$PATH' >> ~/.zshrc
source ~/.zshrc
```

nvm 用户：每次新开终端运行 `nvm use 20` 或写入 `.zshrc`。

</Accordion>

<Accordion title="Codex CLI 怎么升级？">

暂无内置 `codex --upgrade`（issue 仍 open），统一用 npm：

```bash
# npm 全局
npm i -g @openai/codex@latest

# 指定 alpha 版本
npm view @openai/codex versions
npm i -g @openai/codex@0.x.x-alpha.x

# Homebrew 安装的用户
brew upgrade codex
```

</Accordion>

<Accordion title="WSL 下升级 Codex 后无法启动（SQLite 校验失败）">

WSL 环境下 CRLF / LF 行尾差异可能导致 SQLite 迁移校验失败。

```bash
# 备份后删除 state 数据库重新初始化
cp ~/.codex/state_5.sqlite ~/.codex/state_5.sqlite.bak
rm ~/.codex/state_5.sqlite
codex
```

会话历史会丢，但 `config.toml` 和 `auth.json` 保留。

</Accordion>

<Accordion title="VSCode Codex 插件反复 401，但 CLI 正常">

社区高频反馈：相同配置下 CLI 工作正常，VSCode 插件反复 `stream error: exceeded retry limit, last status: 401 Unauthorized`。

**解决顺序**：

1. 在 VSCode 插件中切换为 **"Sign in with ChatGPT"** 方式（很多用户反馈此方式更稳定）。
2. 如继续用 API Key，确保 ProAPI 后台的令牌 **权限为 All**（不是 read-only / restricted），并等待约 5 分钟传播。
3. **不要同时设环境变量和 config.toml**，Key 只放 `~/.codex/auth.json` 即可。
4. 检查 VSCode 插件版本与 CLI 版本是否同步，分别升级。

</Accordion>

<Accordion title="VSCode Codex 插件中加载最新模型">

VSCode Codex 插件内置一份模型列表，若该列表落后于 ProAPI 提供的模型，可以通过替换插件 webview 加载最新模型。

<Tabs items={['Windows', 'macOS']}>
<Tab value="Windows">

1. 打开 VSCode 插件目录：

   ```bash
   %userprofile%\.vscode\extensions
   ```

2. 进入 `openai.chatgpt-<最新版本号>` 目录。

</Tab>
<Tab value="macOS">

1. 在访达按 <kbd>⌘ Command</kbd> + <kbd>Shift</kbd> + <kbd>G</kbd>，输入：

   ```bash
   ~/.vscode/extensions
   ```

2. 进入 `openai.chatgpt-<最新版本号>` 目录。

</Tab>
</Tabs>

3. 依次进入 `webview/assets`，将下载的 **替换脚本** 复制进去。
4. 重启 VSCode，即可在选择器中看到最新模型。

</Accordion>

<Accordion title="登录了 ChatGPT 订阅，但 Codex 一直走 API Key 计费？">

环境变量 `OPENAI_API_KEY` 优先级 **高于** `auth.json` 中的 ChatGPT OAuth，会静默覆盖。修复：

1. `unset OPENAI_API_KEY`（Windows `set OPENAI_API_KEY=`）。
2. 删除 `~/.codex/auth.json` 中的 `OPENAI_API_KEY` 字段。
3. 在 `config.toml` 显式：

   ```toml
   preferred_auth_method = "chatgpt"
   ```

4. 重新 `codex login`。

</Accordion>

<Accordion title="Windows 原生体验差，要不要换 WSL？">

Windows 原生支持仍属「实验性」，存在的严重问题：

- 沙盒 `sandbox-setup.exe` 找不到（[#2794](https://github.com/openai/codex/issues)）
- 启动卡顿 40 秒
- TUI 屏幕渲染错乱（[#13966](https://github.com/openai/codex/issues)）
- Ctrl+C 无法复制
- 中文路径导致乱码
- WSL 与原生 Windows 配置冲突

**官方和社区共识：Windows 用户应使用 WSL2**。在 WSL2 中安装 Node.js 和 Codex CLI，体验与 macOS/Linux 一致。

如必须原生 Windows，在 `config.toml` 中设 `sandbox_mode = "danger-full-access"` 绕过沙盒问题（高风险，谨慎使用）。

</Accordion>
</Accordions>
