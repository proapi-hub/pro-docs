## 章节导读

本页是 Gemini CLI 接入 **ProAPI**（`newapi.prorisehub.com`）时社区反馈最高频的问题合集。涵盖 **中转站与认证类坑**、**模型选择与上下文**、**MCP 与多模态**、**官方版动态与风险** 四大方向。

**基础配置请先阅读**：[Gemini CLI 配置](/cli/gemini) 或 [Gemini 与 Grok 分组](/token/gemini-grok)。

<Callout type="warn" title="现状说明">
  Gemini CLI 当前在第三方中转场景下存在一些已知兼容性问题（多模态、function calling 等）。如果你需要 **稳定使用 Gemini-3**：
  - **首选 Roo Code / Cline 等 VSCode 插件**
  - 必须用 CLI 时，请选用 **Ant-Gemini 分组**渠道
</Callout>

## Gemini CLI 使用现状与建议

**更推荐的接入方式**：

- 优先使用 **Roo Code** 等第三方 VSCode 插件
- 如必须使用 Gemini CLI，建议使用 **Ant-Gemini 分组渠道**（兼容性更好）

<Callout type="info" title="模型名差异">
  Ant-Gemini 分组的 Gemini-3 模型名称可能与官方 API 形式不同，配置前请先到 **模型广场** 确认本分组下的准确模型名。详情查看 [Gemini 与 Grok 分组](/token/gemini-grok)，避免出现「无可用渠道」或「模型不存在」。
</Callout>

**额外提醒**：

- 在 Roo Code 等第三方使用时，请选择 **OpenAI Response** 请求格式。
- Ant-Gemini 分组的模型 **不自带联网**，需要时请通过 MCP 等工具实现。

## 在 Cline 中使用 Gemini-3

### 软件要求

| 软件 | 版本要求 | 下载链接 |
| --- | --- | --- |
| VSCode | 1.80.0+ | [下载 VSCode](https://code.visualstudio.com/) |

### 配置流程

**第一步：创建 Gemini 分组令牌**

按 [创建 API 令牌](/) 一节方法，创建 `Ant-Gemini` 分组令牌并复制 API Key。

**第二步：安装 Cline 插件**

1. 打开 VSCode。
2. 单击左侧 **扩展** 图标（或按 <kbd>Ctrl</kbd> + <kbd>Shift</kbd> + <kbd>X</kbd> / <kbd>⌘ Command</kbd> + <kbd>Shift</kbd> + <kbd>X</kbd>）。
3. 搜索 **Cline**，找到后单击 **安装**。

<Callout type="info" title="安装提示">
  - 安装完成后左侧边栏会出现 Cline 图标。
  - 首次使用需要配置 API Key。
  - 建议安装最新版本以获得最佳体验。
</Callout>

**第三步：打开 Cline 界面**

两种方式：

- **方式一**：单击 VSCode 左侧边栏的 Cline 图标。
- **方式二**：按 <kbd>Ctrl</kbd> + <kbd>Shift</kbd> + <kbd>P</kbd>（macOS 用 <kbd>⌘ Command</kbd>），输入 `Cline: Open` 回车。

**第四步：填写配置**

打开 Cline 后单击 **API Configuration** 按钮，填写：

```yaml
API Provider: OpenAI-compatible
Base URL: https://newapi.prorisehub.com/v1
API Key: sk-*****
Model ID: gemini-3-pro-preview
```

**配置参数说明**：

| 配置项 | 推荐值 | 说明 |
| --- | --- | --- |
| API Provider | `OpenAI-compatible` | 推荐选择此项，支持更多模型 |
| Base URL | `https://newapi.prorisehub.com/v1` | ProAPI 的 OpenAI 兼容端点 |
| API Key | `sk-******` | ProAPI 控制台创建的令牌 |
| Model ID | `gemini-3-pro-preview` | 推荐使用代码专精模型 |

<Callout type="warn" title="安全提醒">
  请妥善保管你的 API Key，不要在群聊或公开截图中泄露。
</Callout>

<Callout type="info" title="已有用户">
  如果之前使用过 Cline，单击右上角 **⚙️ 设置** 按钮进入配置界面修改。
</Callout>

**第五步：完成**

填写完毕后单击右上角 **Done** 即可。

## 中转站与认证类坑

<Accordions>
<Accordion title="GOOGLE_GEMINI_BASE_URL 设了不生效，CLI 还是连官方？">

社区反馈率最高的问题。**根因**：Gemini CLI 有 **两条完全独立的后端路径**，`GOOGLE_GEMINI_BASE_URL` 只在 API Key 路径生效。

**完整配置三步走**：

1. **设置环境变量**（`~/.gemini/.env` 或 shell rc）：

   ```bash
   GEMINI_API_KEY=sk-xxx
   GOOGLE_GEMINI_BASE_URL=https://newapi.prorisehub.com
   ```

   <Callout type="warn" title="末尾不要带 /v1">
     Base URL **不要带** `/v1` 后缀，否则路径拼接错误。
   </Callout>

2. **`settings.json` 显式声明走 API Key 路径**（关键步骤，多数用户漏掉）：

   ```json
   {
     "selectedAuthType": "gemini-api-key"
   }
   ```

   不设这个的话，CLI 默认走 OAuth 路径，会 **完全忽略** `GOOGLE_GEMINI_BASE_URL`。

3. **`.env` 文件位置必须正确**：必须放 `~/.gemini/.env`，**不是项目目录**。

</Accordion>

<Accordion title="同时设了 GEMINI_API_KEY 与 CODE_ASSIST_ENDPOINT，自定义 endpoint 不生效">

Gemini CLI 的两条后端路径完全独立：

| 路径 | 触发条件 | 生效字段 |
| --- | --- | --- |
| API Key 路径 | `.env` 中存在 `GEMINI_API_KEY` | `GOOGLE_GEMINI_BASE_URL` |
| OAuth 路径 | Google 登录 | `CODE_ASSIST_ENDPOINT` |

**`CODE_ASSIST_ENDPOINT` 只在 OAuth 路径下生效**。如果 `.env` 中存在 `GEMINI_API_KEY`，CLI 会跳过 CodeAssist 架构，`CODE_ASSIST_ENDPOINT` 被忽略。

要用代理和自定义 endpoint：**二选一**——

- 删除 `GEMINI_API_KEY` 走 Google OAuth + `CODE_ASSIST_ENDPOINT`。
- 或保留 `GEMINI_API_KEY` + `GOOGLE_GEMINI_BASE_URL`（推荐 ProAPI 用户走这条）。

</Accordion>

<Accordion title="Gemini CLI 报 API key not valid / 401？">

按以下顺序排查：

1. **系统环境变量与 `.env` 冲突** — 检查 shell rc 中是否已 export `GEMINI_API_KEY`：

   ```bash
   echo $GEMINI_API_KEY
   ```

   若存在，删除 rc 中的 export，或修改为与 `.env` 一致。系统环境变量优先级 **高于** `.env`。

2. **`.env` 文件位置错误** — 必须放在 `~/.gemini/.env`，不是项目目录。

3. **令牌分组错误** — Gemini CLI 必须用 **`Ant-Gemini` 分组**的令牌，分组错配是高频原因。

4. **`GOOGLE_GEMINI_BASE_URL` 拼错** — 应为：

   ```bash
   GOOGLE_GEMINI_BASE_URL=https://newapi.prorisehub.com
   ```

   **不要带** `/v1` 后缀。

5. **未设 `selectedAuthType`** — 见上面专题。

</Accordion>

<Accordion title="中转站报 401 / 403 / 404 / model not found，逐项排查">

按 HTTP 状态码定位：

| 状态码 | 含义 | 排查方向 |
| --- | --- | --- |
| **401 Unauthorized** | Key 格式不对，或中转站不认当前认证方式 | 检查 Key 前后空格，确认走 API Key 路径而非 OAuth |
| **403 Permission Denied** | Key 能认证但权限不足 | ProAPI 后台确认令牌权限为 All，确认分组是 Ant-Gemini |
| **404 Not Found** | Base URL 路径错误 | 检查是否需要 `/v1` 后缀（Gemini CLI 走原生协议 **不带**；OpenAI 兼容场景 **带**） |
| **model not found** | 模型 ID 与中转站不一致 | 到 ProAPI [模型广场](/token/intro) 查准确名称，如 `gemini-3-pro-preview` 而非 `gemini-3-pro` |

先用 `curl -I <base_url>` 测试中转站可达性，再排查具体问题。

</Accordion>

<Accordion title="CC-Switch 切换 Gemini 后环境变量没生效">

CC-Switch 在 Google Official 模式下重写 `~/.gemini/.env`，可能 **覆盖用户自定义变量**（如 `GOOGLE_CLOUD_PROJECT`）。

**解决方案**：

- 在 CC-Switch 界面中额外配置所需的环境变量，让 CC-Switch 一并写入。
- 或直接在 shell 配置文件（`.zshrc` / `.bashrc`）中 `export`，不依赖 `.env`，shell 环境变量优先级更高。

</Accordion>

<Accordion title="国内网络无法直连 Gemini CLI">

国内用户直接运行 `gemini` 后，登录卡住或请求超时。三种方案：

| 方案 | 操作 | 适用场景 |
| --- | --- | --- |
| **方案 1**：本地代理 | `gemini --proxy socks5://127.0.0.1:1080` | 本机有梯子 |
| **方案 2**：中转站 | 配置 `GOOGLE_GEMINI_BASE_URL` 指向 ProAPI | **推荐**，零代理零延迟，国内服务器可用 |
| **方案 3**：终端全局代理 | `export https_proxy=http://127.0.0.1:7890` | 临时使用 |

**方案 2 是 ProAPI 用户的标准方案**：不需要代理，国内服务器也能用。

</Accordion>

<Accordion title="Windows 上安装登录卡住、环境变量怎么设？">

Windows 关键差异：

- Node.js 要求 **>= 18**，使用 `npm install -g @google/gemini-cli`
- 环境变量设置方式：
  - CMD：`set GEMINI_API_KEY=sk-xxx`
  - PowerShell：`$env:GEMINI_API_KEY="sk-xxx"`
  - 持久化用「系统环境变量」面板
- `settings.json` 路径：`%USERPROFILE%\.gemini\settings.json`
- **登录卡住** 多半是浏览器回调端口被防火墙拦截，临时关闭防火墙试试

直接走 ProAPI 中转可完全绕过 OAuth 登录流程。

</Accordion>
</Accordions>

## 模型选择与上下文

<Accordions>
<Accordion title="Pro / Flash / Auto / Gemini-3 怎么选？">

| 选项 | 适用场景 | 备注 |
| --- | --- | --- |
| **Auto (Gemini 2.5)** | 大多数用户的默认选择 | 系统根据任务复杂度在 Pro 和 Flash 间自动切换 |
| **Auto (Gemini 3)** | 最新预览版 | 需开启 preview features |
| **Pro 模型** | 复杂调试、架构任务 | 推理更深但消耗更多配额 |
| **Flash 模型** | 简单问答、代码格式转换 | 速度快、成本低 |
| **gemini-3-pro-preview** | ProAPI 中转用户的主力 | Ant-Gemini 分组的预览版名称 |

通过 `/model` 命令切换，或 `--model gemini-2.5-flash` 启动参数指定。

**ProAPI 中转用户注意**：模型 ID **必须与服务商支持的名称完全一致**，否则报 `model not found`。先到 [模型广场](/token/intro) 确认。

</Accordion>

<Accordion title="gemini-3-pro-preview 与 gemini-3-pro 有什么区别？">

- **`gemini-3-pro-preview`** — `Ant-Gemini` 分组当前对外提供的预览版本模型名。
- **`gemini-3-pro`** — Google 官方 API 名称。

ProAPI 的 `Ant-Gemini` 分组以 `preview` 后缀的模型名为准。**模型广场显示的名称才是最终可用的**，不要凭官方文档直接配。

</Accordion>

<Accordion title="thinkingBudget / thinkingLevel 怎么配？响应太慢/费用太高">

不同代别用不同字段：

| 模型代别 | 字段 | 取值 |
| --- | --- | --- |
| Gemini 2.5 系列 | `thinkingBudget` | 数值，0 ~ 32768 |
| Gemini 3 系列 | `thinkingLevel` | `minimal` / `low` / `medium` / `high` |

**重要差异**：

- **Gemini 2.5 Pro 无法关闭思考**（最低 1024）；Flash 可以设 `thinkingBudget: 0` 完全关闭。
- 简单任务建议降到 `thinkingBudget: 1024` 或 `thinkingLevel: "low"`，可显著减少延迟和费用。

在 `settings.json` 的 `modelConfigs.customAliases` 中可自定义思考配置：

```json
{
  "modelConfigs": {
    "customAliases": {
      "fast-pro": {
        "model": "gemini-2.5-pro",
        "thinkingBudget": 1024
      }
    }
  }
}
```

</Accordion>

<Accordion title="100 万上下文窗口但长对话还是会丢信息？">

linux.do 上常见的吐槽：「滑块注意力是纯 shit，上下文长了会漏信息，前面定义的变量后面忘了」。

**根因**：Gemini 标称 1M token 上下文，但实际使用 **滑动窗口注意力**，长上下文质量不稳定。

**建议**：

- 编码任务 **拆分为小模块**，不要依赖单次超长对话。
- 使用 `/compress` 命令压缩当前对话上下文。
- 关键决策写入 `GEMINI.md` 而非聊天历史。
- Gemini 更适合 **前端设计**（"有灵气"），不适合写需要严格一致性的后端代码。
- 严肃后端开发场景，社区共识：换 Claude Code。

</Accordion>

<Accordion title="Gemini CLI vs Claude Code 怎么选？">

社区共识：

| 维度 | Gemini CLI | Claude Code |
| --- | --- | --- |
| **免费额度** | 大 | 无免费 CLI |
| **上下文窗口** | 1M token | 200K（可开启 1M） |
| **代码质量** | 一般 | **更高** |
| **多文件重构** | 容易出错 | **可靠** |
| **跨文件理解** | 浅 | **深** |
| **权限模型** | 弱 | **完善** |
| **多模态** | **原生强** | 不支持 PDF |
| **内置搜索** | **有** | 无 |
| **开源** | **是** | 否 |

**实战建议**：

- **快速原型、文档分析、多模态** → Gemini CLI
- **精确代码质量、严肃工程** → Claude Code
- **同时装两个**，根据场景切换，是当前主流配置

</Accordion>

<Accordion title="GEMINI.md 项目级提示词如何使用？">

类似 Codex 的 `AGENTS.md`，Gemini CLI 支持在项目根创建 `GEMINI.md` 作为项目级提示词。加载顺序：

1. `~/.gemini/GEMINI.md`（全局）
2. 项目根 `GEMINI.md`
3. 当前 cwd `GEMINI.md`（按需补充）

子目录的 `GEMINI.md` **补充而不覆盖** 上级目录的内容。

**调试命令**：

- `/memory show` — 查看当前加载了哪些上下文文件
- `/memory refresh` — 重新扫描并加载
- `contextFileName` — 在 `settings.json` 中可自定义文件名

</Accordion>

<Accordion title="免费额度突然变成 limit: 0？">

Gemini 免费层有几个分级，遇到 429 / `limit: 0` 排查：

| 层级 | 配额 | 触发 limit: 0 的原因 |
| --- | --- | --- |
| Google 账号登录 | 1000 次/天 | 与 Gemini Code Assist 智能体模式共享 |
| API Key 免费层 | 250 次/天 | **未关联 billing**（即使免费层也需要绑定结算账号） |
| Google AI Pro ($25/月) | 1500 次/天 | 升级方案 |

`limit: 0` 的常见原因是 Google Cloud 项目 **未启用结算账号**。即使只用免费层，也需要在 Google Cloud Console 中关联一个 billing 账号。

直接走 ProAPI 中转可绕过 Google 配额限制，**按量计费、无每日上限**。

</Accordion>

<Accordion title="Gemini-3 的 thinking / safetySettings 通过 OpenAI 兼容协议会丢失？">

会。OpenAI 兼容协议不识别这些 Gemini 原生字段：

- `thinking_budget` — 思考 token 上限
- `safetySettings` — 4 类风险过滤阈值
- `systemInstruction` — 独立的系统提示

部分客户端（Cline 等）可用 `extra_body` 传递 Gemini 特有参数：

```json
{
  "extra_body": {
    "thinkingConfig": { "thinkingBudget": 8192 }
  }
}
```

**如果必须使用这些字段**，请走 [Gemini 原生端点](/api/gemini)：

```text
POST https://newapi.prorisehub.com/v1beta/models/{model}:generateContent
```

</Accordion>
</Accordions>

## MCP 与多模态

<Accordions>
<Accordion title="MCP 服务器配置了但 /mcp list 显示未就绪？">

逐项检查：

1. **`command` 路径正确，可在终端中直接运行** — 用 `which <command>` 验证。
2. **优先用 `gemini mcp add` 命令** 而不是手动编辑 JSON，命令行能帮你校验。
3. **`env` 中可以用 `$VAR_NAME` 引用系统变量**：

   ```json
   {
     "env": {
       "GITHUB_TOKEN": "$GH_TOKEN"
     }
   }
   ```

4. **远程 MCP** — `url` 或 `httpUrl` 需要确保网络可达。
5. **`/mcp refresh`** 重启 MCP 服务器；**`/mcp desc`** 获取工具详细描述。

</Accordion>

<Accordion title="为什么 Gemini CLI 不能粘贴图片 / 多模态输入失效？">

Gemini CLI 的多模态当前 **不完全支持剪贴板**。三种替代方案：

**方案 1：使用 `@` 引用图片文件**

```text
@./screenshot.png 帮我描述这张图
```

支持 PDF、音频、视频。

**方案 2：尝试 `Ctrl+V` 粘贴**

最新版本部分场景可用，但成功率不稳定。

**方案 3：改用 VSCode 插件**

- 改用 **Cline / Roo Code** 等接入 Gemini-3，剪贴板支持完整。
- 或在 [API 端点章节](/api/gemini) 直接通过 `inlineData` 走 `/v1beta/.../generateContent` 提交 base64 图片。

</Accordion>

<Accordion title="Cline 用 Gemini 报 'Provider does not support function calling'？">

OpenAI 兼容协议下 Gemini 的 function calling 字段被翻译时可能丢失。解决方案：

1. **关闭 Cline 的 MCP 调用**（在「MCP Servers」侧栏禁用所有 server），让 Gemini 走纯文本对话。
2. **切到 Roo Code** — Roo Code 的 OpenAI Response 协议对 Gemini function calling 兼容更好。
3. **改用更轻的任务** — Gemini 走 OpenAI 兼容时较适合纯代码补全和解释，不适合需要密集工具调用的 agent 场景。

</Accordion>

<Accordion title="Roo Code 与 Cline 是什么关系？">

Roo Code 是 Cline 的 fork，相比 Cline 增加：

- **自定义 Mode** — 可定义 Architect、Code、Ask 等专用角色。
- **Orchestrator** — 多 mode 协作的编排器。
- **OpenAI Response 请求格式** — 对 GPT-5 Responses API 与 Gemini-3 思考字段兼容更好。

如果 Cline 出现 function calling 兼容问题，优先尝试 Roo Code。

</Accordion>

<Accordion title="Roo Code 找不到 'OpenAI Response' 选项？">

旧版本 Roo Code 仅有 `OpenAI Compatible`。请升级：

1. VSCode 扩展面板搜索 Roo Code。
2. 单击「更新」到最新版（≥ 3.x）。
3. 重启 VSCode。

升级后 `API Provider` 下拉中会出现 **OpenAI Response** 选项。

</Accordion>

<Accordion title="第三方 CLI 用 Ant-Gemini 时如何启用联网搜索？">

Ant-Gemini 分组的模型 **不自带联网**。需要联网时：

1. **通过 MCP** — 接入 `tavily-mcp` 或 `brave-search-mcp` 等搜索 MCP。
2. **改用 Grok 分组** — Grok 模型自带搜索能力，当前默认倍率为 `x0.1`。
3. **改用 `chub` Context Hub MCP** — ProAPI 推荐的文档检索方案。

</Accordion>

<Accordion title="Gemini CLI 内置的 google_web_search / web_fetch 怎么用？">

走 Google 官方账号路径时，CLI 内置两个工具：

- **`google_web_search`**：直接在提示中提问实时信息，CLI 自动调用搜索。
- **`web_fetch`**：用自然语言要求抓取指定 URL 的内容并总结。

使用时 CLI 会请求权限确认，可选「允许一次」或「始终允许」。`--yolo` 模式可跳过权限确认（**不推荐**）。

用 `/tools` 命令查看所有可用工具。

**走 ProAPI 中转时这两个工具不可用**，请走 MCP 方案。

</Accordion>

</Callout>

</Accordion>

<Accordion title="如何在 Gemini CLI 中切换主题与认证方式？">

Gemini CLI 支持下列常用配置项（写入 `~/.gemini/settings.json`）：

```json
{
  "theme": "Dark",
  "model": "gemini-2.5-pro",
  "selectedAuthType": "gemini-api-key"
}
```

- `theme` — 内置主题如 `Dark` / `Light` / `Dracula` / `GitHub` / `Google Code`。`/theme` 命令交互式切换。
- `model` — 默认模型，与 `.env` 中的 `GEMINI_MODEL` 二选一即可。
- `selectedAuthType` — 第三方中转使用 `gemini-api-key`，官方账号使用 `oauth-personal`。

<Callout type="warn">
  如果 `settings.json` 中定义了 theme，必须先删除才能用 `/theme` 命令切换。
</Callout>

</Accordion>
</Accordions>

## 官方版动态与风险

<Accordions>
<Accordion title="2026 年 6 月 18 日起 Gemini CLI 免费版迁移到 Antigravity CLI">

Google 官方公告：**2026-06-18 起**，Gemini Code Assist IDE 扩展和 Gemini CLI 将停止为下列群体提供服务：

- 个人版（免费）
- Google AI Pro 订阅
- Google AI Ultra 订阅

需要迁移到 **Antigravity** 和 **Antigravity CLI**。

**保留访问权限**：

- Gemini Code Assist **Standard** 订阅者
- Gemini Code Assist **Enterprise** 订阅者

**对 ProAPI 用户的影响**：**零影响**。ProAPI 走的是 Gemini API 而不是 Code Assist 后端，迁移与你无关。继续走 ProAPI 中转即可。

</Accordion>

<Accordion title="用第三方工具反代 Gemini CLI 的 OAuth 会被封号？">

部分社区工具（如 CLIProxyAPI、OpenClaw、OpenCode）会把 Gemini CLI 的 OAuth 登录态转为 API 接口供其他客户端使用。

<Callout type="warn" title="Google 官方明确警告">
  使用第三方软件访问 Gemini CLI 的免费配额 **违反服务条款**，可能导致账号被 **暂停或终止**。
</Callout>

这些工具技术上可用，但封号风险真实存在。**Google 官方建议**：如需第三方编码代理使用 Gemini，请使用 **Vertex AI** 或 **AI Studio API 密钥**。

**ProAPI 用户无需担心**：你走的是按量付费的 API 密钥路径，不是 OAuth 反代，不违反任何条款。

</Accordion>

<Accordion title="--yolo 模式的安全风险">

`--yolo` 跳过所有工具调用的权限确认，**包括文件写入、shell 命令执行**。社区已有用户因此被 Gemini CLI 自动执行 `rm -rf`、误删数据库等。

Google 官方文档明确「**通常不建议使用**」。

**更安全的替代**：

- 使用 `--sandbox docker` 或 `--sandbox podman` 在沙盒中执行。
- `settings.json` 的 `excludeTools` 排除危险工具：

  ```json
  {
    "excludeTools": [
      "run_shell_command(rm)",
      "run_shell_command(sudo)"
    ]
  }
  ```

- 也可用 `Ctrl+Y` 临时切换 YOLO 模式。

</Accordion>

<Accordion title="非交互模式 CI/CD 集成">

想在脚本或自动化流程中使用：

```bash
# -p 参数或直接传位置参数
gemini -p "What is the gcloud command to deploy to Cloud Run"
gemini "Summarize the latest commit"
```

**限制**：

- 非交互模式下 **无法追问问题**。
- **无法授权工具调用**（包括 WriteFile）。
- 适合简单的一次性查询，不适合需要工具调用的复杂任务。
- 可结合 shell 管道：

  ```bash
  cat logs.txt | gemini -p "Find errors"
  ```

</Accordion>

<Accordion title="Gemini CLI 中文汉化版能用吗？">

社区有 `gemini-cli-chinese`（GitHub: `396001000/gemini-cli-chinese`）汉化版，新增 `/baseurl`、`/setkey`、`/status` 等中文命令，内置中转站配置能力。

**风险**：

- 与官方版本可能存在 **版本落后** 问题。
- 第三方维护，**安全性自行评估**。
- 接入新模型可能需要等待汉化版作者同步。

**ProAPI 用户建议**：直接用官方版 + 正确的 `settings.json` 配置，参考 [Gemini CLI 配置](/cli/gemini) 即可，不必依赖汉化版。

</Accordion>
</Accordions>
