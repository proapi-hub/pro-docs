## 章节导读

本页是 Claude Code 接入 **ProAPI**（`newapi.prorisehub.com`）时社区反馈最高频的问题合集。涵盖 **中转站配置**、**Prompt Cache 与费用优化**、**上下文管理**、**Windows 与安装**、**多供应商切换** 五大方向。

**基础配置请先阅读**：[Claude Code 配置](/cli/claude-code) 或 [CC-Switch 接入](/ccswitch/claude-code)。

## 配置指南

### VSCode Claude Code 插件强制登录绕过

Claude Code `2.0` 起 VSCode 插件默认强制 OAuth 登录，第三方中转站用户必须手动绕过。

<Tabs items={['Windows', 'macOS']}>
<Tab value="Windows">

1. 确保你已完成 [环境检查](/cli/env-check)，且 `claude` CLI 已能正常对话。
2. 按 <kbd>Win</kbd> + <kbd>R</kbd>，输入：

   ```bash
   %userprofile%\.claude
   ```

3. 如目录中没有 `config.json`，请手动创建：

   ```json
   {
     "primaryApiKey": "ProApi"
   }
   ```

   <Callout type="info">
     `primaryApiKey` 的值可为任意字符串，只是用于跳过强制登录检查。真正的 Key 在 `settings.json` 的 `env` 块中配置。
   </Callout>

4. 重启 VSCode 后插件即可使用 ProAPI。

</Tab>
<Tab value="macOS">

1. 确保你已完成 [环境检查](/cli/env-check)，且 `claude` CLI 已能正常对话。
2. 在访达中按 <kbd>⌘ Command</kbd> + <kbd>Shift</kbd> + <kbd>G</kbd>，输入：

   ```bash
   ~/.claude
   ```

3. 如目录中没有 `config.json`，请手动创建：

   ```json
   {
     "primaryApiKey": "ProApi"
   }
   ```

4. 重启 VSCode 后插件即可使用 ProAPI。

</Tab>
</Tabs>

### Claude Code 无法连接到 Anthropic 服务

<Callout type="info" title="首次配置时直接照做">
  如果你是首次配置被引导到这里，请直接按下方命令操作即可。
</Callout>

使用 npm 安装完 `claude` 后启动出现：

```text
Unable to connect to Anthropic services
Failed to connect to api.anthropic.com: ERR BAD REQUEST
Please check your internet connection and network settings.
```

或要求登录 Anthropic 账号。原因是 Claude Code 默认要求 onboarding 流程，需手动跳过：

<Tabs items={['Windows', 'macOS']}>
<Tab value="Windows">

1. 按 <kbd>Win</kbd> + <kbd>R</kbd>，输入 `cmd` 打开命令行。
2. 运行：

   ```bash
   powershell -Command "$f='%USERPROFILE%\.claude.json';$j=Get-Content $f|ConvertFrom-Json;$j|Add-Member -NotePropertyName 'hasCompletedOnboarding' -NotePropertyValue $true -Force;$j|ConvertTo-Json|Set-Content $f"
   ```

3. 重启 Claude Code CLI。

</Tab>
<Tab value="macOS">

1. 打开「终端」程序。
2. 运行（依赖 `jq`）：

   ```bash
   jq '. + {"hasCompletedOnboarding": true}' ~/.claude.json > /tmp/tmp.json && mv /tmp/tmp.json ~/.claude.json
   ```

   <Callout type="info" title="未安装 jq">
     提示 `command not found: jq`？运行 `brew install jq` 安装。
   </Callout>

3. 重启 Claude Code CLI。

</Tab>
</Tabs>

### 切换回 200K 上下文并禁用非必要流量

ProAPI 用户推荐的 `settings.json.env` 基线：

```json
{
  "env": {
    "CLAUDE_CODE_DISABLE_1M_CONTEXT": "1",
    "CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC": "1",
    "CLAUDE_CODE_DISABLE_TERMINAL_TITLE": "1",
    "CLAUDE_CODE_ATTRIBUTION_HEADER": "0",
    "DISABLE_TELEMETRY": "1",
    "DISABLE_ERROR_REPORTING": "1"
  }
}
```

每一项的作用与必要性，详见后文 [常见问题 → 费用与 Prompt Cache](#费用与-prompt-cache-优化)。

| `/model` / `/cost` / `/usage` | 切换模型 / 查看本次费用 / 查看额度使用 |

## 费用与 Prompt Cache 优化

<Callout type="warn" title="单条最重要的配置">
  接入 ProAPI 中转 API 时，**`CLAUDE_CODE_ATTRIBUTION_HEADER=0` 是单条收益最高的配置**。开启前后实测：单次费用降 90%、首包延迟从 17 秒回到 2 秒。详见下方第一个折叠项。
</Callout>

<Accordions>
<Accordion title="为什么必须设置 CLAUDE_CODE_ATTRIBUTION_HEADER=0？">

Claude Code `2.1.36` 起会在每次请求的 system prompt 开头注入动态归因标识：

```text
x-anthropic-billing-header: cc_version=2.1.xx.abc; cc_entrypoint=claude-code; cch=00000;
```

其中 `cc_version` 后缀（如 `.abc`）是根据当前会话的首条用户消息计算出的哈希。**每开一个新会话，这段文本就会变化**，导致：

- 庞大的 system prompt（约 6.8 万 tokens）**前缀** 每次都不一样
- 中转 API 的 Prompt Cache **无法命中**，每次都要完整处理 system prompt
- 实测费用最高暴涨 10 倍、首包延迟从 2 秒变成 17 秒

**关闭前后对比**：

| 指标 | 关闭前 | 关闭后 |
| --- | --- | --- |
| Cache Read tokens | 512 | 68,096 |
| 单次费用 | 1× 基准 | **约 0.1×（降 90%）** |
| API 首包延迟 | ~17s | ~2s |

**有什么副作用？** 没有。这个字段只是 Anthropic 用于内部归因统计的标签，关闭后不影响任何功能、模型行为或响应质量。

<Callout type="warn" title="CC-Switch 用户也需要手动配置">
  截至当前版本，CC-Switch **不会自动注入** 此环境变量（[功能请求 #2025](https://github.com/farion1231/cc-switch/issues/2025) 仍在 open 状态）。请在 CC-Switch「环境变量」编辑框中手动追加 `CLAUDE_CODE_ATTRIBUTION_HEADER=0`。
</Callout>

</Accordion>

<Accordion title="为什么我的额度消耗这么快？只用了 20 分钟就跑完">

额度不按时间线性扣，与 **模型、上下文长度、附件、代码库规模、对话历史、工具调用** 都相关。常见原因排行：

1. **全程用 Opus 处理简单任务** — Opus 价格是 Sonnet 的 **5 倍**，日常编码用 Sonnet 就够。
2. **CLAUDE.md 写太长** — 每条消息都带上几千 token 的项目说明，控制在 200 行以内。
3. **MCP Server 全开** — 每个 MCP 工具定义都会塞进 system prompt，只启用当前需要的 5-8 个。
4. **任务完成后不 `/clear`** — 历史会一直累积；阶段完成后用 `/clear` 重新开。
5. **粘大日志或大文件** — 100KB 的日志整包塞入直接吃掉 25K token，只贴关键片段。
6. **频繁切模型/切 settings** — 破坏 prompt cache，每次都按全价计算。

用 `/cost` 实时查看本次费用，`/context` 查看上下文占用比例。

</Accordion>

<Accordion title="Prompt Cache 实际怎么用？为什么我感觉没省钱">

Cache 命中的 token 只收正常价的 **10%**（Sonnet：$0.30/MTok vs $3/MTok），但很多用户由于以下原因导致命中率极低：

- **会话闲置超过 1 小时** — 订阅版 cache TTL 为 1 小时（API 版仅 5 分钟），过期后重新发请求整段 transcript 走 `cache_creation`（单价是 `cache_read` 的 10-12 倍）。
- **中途用 `/model` 切换模型** — 整个缓存前缀全部失效，下一轮变成 `cache_creation`，明显感觉变慢变贵。
- **频繁修改 CLAUDE.md / 启停 MCP / 改 system prompt** — 前缀变化即缓存失效。
- **未设置 `CLAUDE_CODE_ATTRIBUTION_HEADER=0`** — 见上一题，归因 header 让 system prompt 前缀每次都不同。
- **subagent cache TTL 永远是 5 分钟** — 长任务 subagent 容易丢失缓存。

**最佳实践**：保持 CLAUDE.md 稳定、不要中途切模型、长时间离开前先 `/compact` 而非闲置。

</Accordion>

<Accordion title="Sonnet / Opus / Haiku 怎么分工最省钱？">

社区共识的「三层模型分工」：

| 模型 | 占比建议 | 适用场景 |
| --- | --- | --- |
| **Sonnet 4.6**（默认） | 80% | 日常编码、改 bug、阅读代码、小型重构 |
| **Opus 4.7/4.8** | 15% | 跨文件大重构、架构设计、疑难推理、多步规划 |
| **Haiku 4.5** | 5% | 代码格式化、写注释 / commit message、文件搜索、简单查询 |

在 `~/.claude/settings.json` 中配置默认与小模型：

```json
{
  "env": {
    "ANTHROPIC_MODEL": "claude-sonnet-4-6",
    "ANTHROPIC_SMALL_FAST_MODEL": "claude-haiku-4-5-20251001",
    "ANTHROPIC_DEFAULT_HAIKU_MODEL": "claude-haiku-4-5-20251001"
  }
}
```

会话内随时 `/model opus` 切换。**判断口诀**：写代码 → Sonnet；读 log / 格式化 → Haiku；改架构 → Opus。

实际可用模型名以 ProAPI [模型广场](/token/intro) 为准。

</Accordion>

<Accordion title="CLAUDE.md 写多长合适？感觉写了规则 Claude 也不遵守">

社区经验：CLAUDE.md 控制在 **200 行以内**（约 500-800 token）。写太长会出现两个问题：

1. **每次对话都带上几 KB 上下文**，长期开销巨大。
2. **规则越多遵从度越低** — Claude 在过载规则下会开始"挑着遵守"。

**精简法则**：

- 只放「每次都适用的核心规则」（项目结构、代码风格、关键约束）。
- 偶尔用的规则移到 `.claude/skills/<name>.md` 或 slash command。
- 删掉 Claude 本来就知道的通用最佳实践（如「写清晰的命名」）。
- 每条规则问自己：**不写这条 Claude 会犯这个错吗**？不会就删。

用 `/memory` 查看实际加载了哪些规则文件。

</Accordion>
</Accordions>

## 上下文管理进阶

<Accordions>
<Accordion title="长对话后 Claude 开始前后矛盾、给的回答越来越差？">

经典「上下文衰退」症状，200K 窗口填满后性能会显著下降。处理顺序：

1. **60-70% 时主动 `/compact`** — 不要等到自动压缩（约 83.5% 时才触发，此时质量已经下降）。
2. **同一问题纠正 2 次以上立即 `/clear`** — 比修小错收益更大。
3. **重要决策用 `#` 写进 CLAUDE.md** — 跨会话有效，不依赖压缩存活。
4. **大范围探索委托给 subagent** — 探索结果只回主对话一个摘要，不污染主上下文。
5. **拆任务** — 单会话超过 3 小时，质量曲线开始走低。

</Accordion>

<Accordion title="/compact 后关键信息被压缩掉了怎么办？">

`/compact` 默认会按重要性压缩，但模型对「重要」的判断未必和你一致。**正确用法**：传入具体保留指令：

```text
/compact 保留 API 规范、认证流程的架构决策、已确认的字段格式
```

进阶做法：

- **压缩前先把重要结论写进 CLAUDE.md**，压缩后仍可读取。
- 在 CLAUDE.md 中加 `## Compact Instructions` 段，指定每次压缩必须保留的内容。
- 调试失败的尝试不值得保留，可以更激进地压缩。

</Accordion>

<Accordion title="子目录的 CLAUDE.md 不生效？">

这是个常见误解：**子目录 CLAUDE.md 是按需加载的，不是会话开始时全部加载**。

- 只有 Claude 用 Read 工具读取该目录中的文件时，对应 CLAUDE.md 才会被加载。
- 内置的 **Explore** 和 **Plan** 子代理会 **跳过** CLAUDE.md。
- 必须全局生效的关键规则，放在 **项目根 CLAUDE.md** 或 **`~/.claude/CLAUDE.md`**（全局）。
- 用 `/memory` 命令实时查看当前会话加载了哪些 CLAUDE.md。

</Accordion>

<Accordion title='看到 "Autocompact is thrashing" 怎么处理？'>

上下文被某个大文件 / 大工具输出反复填满，自动压缩压不下去。处理：

- **改成分段读文件** — 指定行号范围 `Read offset=100 limit=200`，不要一次读整个文件。
- **手动激进压缩** — `/compact keep only the plan and the diff`。
- **大量文件检索丢给 subagent** — 主对话只接收摘要。
- **`/clear` + 精简提问重启** — 当压缩已经收效甚微。

</Accordion>

<Accordion title="如何用 subagent 节省 90% 的探索 token？">

让 Claude 扫描大量文件研究问题时，所有文件内容都进入主上下文，迅速填满。**替代方案**：

```text
使用子代理分析 logs/ 目录下所有 .log 文件，只把结论告诉我
```

Subagent 在独立 context window 中运行，最后只回摘要。**实测**：6100 token 的文件读取 → 约 420 token 的摘要，节省 **超过 90%**。

适用场景：代码库探索、代码审查、文档搜索、bug 调查、依赖分析。

</Accordion>

<Accordion title=".claudeignore 是被低估的一次性优化">

Claude Code 在探索代码库时会读 `node_modules`、`dist`、`.next` 等无关文件，浪费 token。**投资 5 分钟创建 `.claudeignore`，每次 session 节省 20-40% token**。

推荐模板：

```text
node_modules/
dist/
build/
.next/
.turbo/
.cache/
coverage/
*.lock
package-lock.json
pnpm-lock.yaml
yarn.lock
*.log
.DS_Store
.env*
*.min.js
*.min.css
```

语法与 `.gitignore` 完全相同，放在仓库根目录即可。

</Accordion>

<Accordion title="100% context used 后报 524 / API Error / 无限重试 attempt N/10">

经典故障链：

1. 会话 `100% context used`，启用了 thinking
2. 模型 prefill + thinking 阶段长达 2-5 分钟，**期间没有任何字节返回**
3. 经过 CDN 的链路触发 `Cloudflare Error 524 / origin_response_timeout`
4. Claude Code 把 524 当作 `retryable`，进入 `attempt 2/10`、`3/10` 循环
5. 每次重试上下文还是满的，反复撞同样的墙

**根因不在你这边**：Cloudflare 边缘默认 100 秒 `proxy_read_timeout`（Free/Pro/Business 计划均为硬限制，不可调）。SSE 流式响应即使已经返回 200，只要 100 秒内没有字节经过边缘，CDN 就会切断。

**用户侧能立刻做的**：

| 优先级 | 操作 | 效果 |
| --- | --- | --- |
| ★★★ | 看到 `attempt 2/10` 立刻 <kbd>Esc</kbd> 中断 | 避免重试雪崩 |
| ★★★ | `/compact 保留关键决策与未完成 TODO` | 直接消除长 TTFT 根源 |
| ★★★ | `settings.json.env` 设 `"CLAUDE_CODE_DISABLE_1M_CONTEXT": "1"` | 回退 200K 窗口，prefill 时间砍半 |
| ★★ | 复杂 thinking 任务换 **Sonnet** 而非 Opus | Opus thinking 阶段显著更长 |
| ★★ | 单会话不要堆超过 100K token；阶段完成后 `/clear` | 从源头避免 100% context |
| ★ | 设 `CLAUDE_CODE_MAX_THINKING_TOKENS` 限制思考预算 | 控制单次 thinking 长度 |

**已经 100% 卡住怎么办**：

1. <kbd>Esc</kbd> 强制中断当前请求。
2. `/compact 保留 ...` 显式指定要保留的关键信息。
3. 若 `/compact` 也撞超时，`/clear` 后从 CLAUDE.md 重新装载关键决策。
4. 若长期反复出现，反馈到 ProAPI 群内，附 `ray_id` 与 `instance` 字段（截图里 524 错误体中有），便于服务端定位。

<Callout type="info" title="服务端侧也在持续优化">
  我们已知该问题与 Cloudflare 边缘 100 秒 read timeout 相关，正在通过 SSE keep-alive 心跳改造彻底解决。用户侧上述策略是「即时止血」，不依赖服务端修复也能跑通。
</Callout>

</Accordion>
</Accordions>

## Windows 与安装新坑

<Accordions>
<Accordion title='Windows 安装报"不支持 16 位应用"/启动失败？'>

Anthropic 已将 Claude Code 切换为 **原生安装方式**，npm 包已被官方标记为 deprecated（自动更新后产物结构不兼容旧版本运行时）。

**推荐方案**：

```powershell
# 卸载 npm 版
npm uninstall -g @anthropic-ai/claude-code

# 官方安装器（推荐）
irm https://claude.ai/install.ps1 | iex

# 或用 winget
winget install Anthropic.ClaudeCode
```

安装到 `~/.local/bin/claude`（macOS/Linux）或 `%USERPROFILE%\.local\bin\claude.exe`（Windows）。

</Accordion>

<Accordion title="Windows 中文用户名导致 ENOENT 安装失败">

用户名含中文（如 `C:\Users\神鹰`）时，npm 全局安装路径含 Unicode 字符，触发 `ENOENT`。

**方案 1**：为 npm 指定纯英文全局目录：

```powershell
npm config set prefix "C:\npm-global"
# 将 C:\npm-global 添加到系统 PATH
```

**方案 2**：直接用上一题的 **原生安装器**，完全绕过 npm。

</Accordion>

<Accordion title="Windows /copy 复制中文显示乱码">

根因是 `clip.exe` 使用系统 OEM 代码页（CP936），不支持 UTF-8。

- **2026-03-10 起官方版本已修复**，运行 `claude update` 更新即可。
- 如必须用旧版的补丁方案：修改 `cli.js` 中 `windows:["clip"]` 为 `windows:["chcp 65001 >nul & clip"]`。

</Accordion>

<Accordion title='CMD / PowerShell 显示乱码、输入框越界？'>

按以下顺序处理：

1. 改用 **Windows Terminal + PowerShell 7**（强烈推荐）。
2. 终端中执行 `chcp 65001` 切到 UTF-8。
3. 或直接在 **WSL2** 内安装运行 `claude`，规避 Windows 终端兼容性问题。
4. 升级到��新 `claude` 版本：`claude update`。

</Accordion>

<Accordion title="claude 突然闪退、卡死十几分钟后才退出？">

近期版本的两个已知 bug：

- **`v2.1.89` Bun GC 崩溃** — 表现为卡住 10-20 分钟后退出。
- **`v2.1.83+` 大仓库 OOM** — 在超大型 monorepo 上吃光内存。

**处理顺序**：

1. 运行 `/doctor` 诊断（会输出版本与环境信息）。
2. 临时回滚到稳定版：`npm install -g @anthropic-ai/claude-code@2.1.81`（如还在用 npm 版）。
3. 大仓库 OOM：配置 `.claudeignore` 排除无关目录，增加系统 swap。
4. 多重安装冲突：`which -a claude` 检查是否同时存在 npm 版和 native 版，删掉旧的。

</Accordion>

<Accordion title="WSL 下性能极慢？">

最常见原因是把代码放在 Windows 文件系统（`/mnt/c/...`）下，跨文件系统 IO 极慢。

- **代码仓库放到 WSL 自己的文件系统** —— `~/projects/...` 而非 `/mnt/c/...`。
- 确认 `which node` / `which npm` 指向 Linux 版本，而不是 `/mnt/c/Program Files/...` 下的 Windows 版本（会被自动加入 PATH）。
- WSL2 中 OAuth 登录时浏览器不会自动打开，需要手动复制终端中的 URL 到 Windows 浏览器打开。

</Accordion>

<Accordion title="企业证书 / 自签证书导致 unable to verify the first certificate？">

公司网络常见。导出企业根证书为 PEM，然后：

```bash
export NODE_EXTRA_CA_CERTS=/path/to/ca.pem
```

macOS 上 Claude Code 2.1.17+ 改用 Bun，需同时加：

```bash
export NODE_OPTIONS="--use-openssl-ca"
```

写入 `~/.zshrc` 或 `~/.bashrc` 持久化。mTLS 场景额外配置：

```bash
export CLAUDE_CODE_CLIENT_CERT=/path/to/client.pem
export CLAUDE_CODE_CLIENT_KEY=/path/to/client.key
```

</Accordion>
</Accordions>

## 多供应商与切换

<Accordions>
<Accordion title="ANTHROPIC_API_KEY 和 ANTHROPIC_AUTH_TOKEN 到底用哪个？">

二选一关系，**不能共存**。区别如下：

| 变量 | 适用场景 | 优先级 |
| --- | --- | --- |
| `ANTHROPIC_API_KEY` | 标准 Anthropic 兼容接口（如月之暗面 K2、官方 API） | 高于 OAuth |
| `ANTHROPIC_AUTH_TOKEN` | **中转站**（ProAPI、NewAPI/One-API 类） | 不与 OAuth 冲突 |

ProAPI 用户使用 **`ANTHROPIC_AUTH_TOKEN`**。所有配置都建议放在 `~/.claude/settings.json` 的 `env` 块中，不要改系统环境变量（污染面更大、清理更难）。

</Accordion>

<Accordion title="第三方 API 报 401 Invalid bearer token？">

配置了 `ANTHROPIC_BASE_URL` + `ANTHROPIC_AUTH_TOKEN` 后仍 401，按以下顺序排查：

1. 系统环境变量与 `settings.json` 中是否同时存在 `ANTHROPIC_API_KEY` 与 `ANTHROPIC_AUTH_TOKEN`，删除多余的。
2. 之前登录过官方 OAuth？执行 `/logout` 后重启。
3. `~/.claude.json` 中残留旧 OAuth 信息，清理 `oauthAccount` 字段后重试。
4. 确认 `ANTHROPIC_BASE_URL` 末尾 **不要带** `/v1`。
5. ProAPI 后台「令牌管理」确认 Key 状态正常、且分组为 Claude Code 专用分组。

</Accordion>

<Accordion title="如何用 --settings 优雅切换多个中转站？">

Claude Code 原生支持 `--settings` 参数，比 CC-Switch 更轻量。**推荐工作流**：

按供应商准备多份 settings：

```text
~/.claude/settings.proapi.json
~/.claude/settings.kimi.json
~/.claude/settings.glm.json
```

每个文件独立 `env` 配置，启动时指定：

```bash
claude --settings ~/.claude/settings.proapi.json
```

配合 shell 别名：

```bash
alias cc-pro="claude --settings ~/.claude/settings.proapi.json"
alias cc-kimi="claude --settings ~/.claude/settings.kimi.json"
```

切换零成本、零残留。

</Accordion>

<Accordion title="切换 CC-Switch 或多供应商后旧 token 仍然生效？">

依次清理：

1. 临时清空环境变量：

   ```bash
   unset ANTHROPIC_API_KEY ANTHROPIC_AUTH_TOKEN ANTHROPIC_BASE_URL
   ```

2. 删除 shell rc（`.zshrc` / `.bashrc`）中残留的 `export ANTHROPIC_*`。
3. 编辑 `~/.claude.json`，移除 `oauthAccount` 与项目级 `apiKeyHelper` 字段。
4. 重启终端再次运行 `claude`。

</Accordion>

<Accordion title="挂了一晚后突然全部 401？">

可能原因与对策：

1. **配置变更未生效** — `settings.json` 修改后需重启 CLI 进程。
2. **OAuth 会话过期** — 执行 `/logout` 再 `/login`，或检查 `~/.claude.json` 中 `oauthAccount`。
3. **令牌被禁用** — 到 ProAPI 后台「令牌管理」检查状态与余额。
4. **本机时间漂移** — 校准系统时间，TLS 握手对时间敏感。

</Accordion>

<Accordion title="如何添加 MCP 服务器？">

接入 GitHub / Filesystem / Context7 等 MCP，推荐用 `claude mcp add`：

```bash
# HTTP 远程
claude mcp add --transport http context7 https://mcp.context7.com/mcp

# 本地 stdio
claude mcp add --transport stdio github -- npx -y @modelcontextprotocol/server-github
```

作用域选项：

- `--scope user` — 跨项目共享，写入 `~/.claude.json`。
- `--scope project` — 写入仓库 `.mcp.json`，团队共享。

<Callout type="warn" title="MCP 工具数与 token 消耗">
  每个 MCP 工具定义都会塞进 system prompt。**建议同时启用不超过 5-8 个**，用 `/mcp` 命令按需启停。
</Callout>

</Accordion>

<Accordion title="permissions.allow / deny 不生效？">

- **文件位置**：个人放 `.claude/settings.local.json`，团队共享放 `.claude/settings.json`。
- **评估顺序**：`deny → ask → allow`，deny 始终优先。
- **语法示例**：

  ```json
  {
    "permissions": {
      "allow": ["Bash(git *)", "Edit(./src/**)"],
      "deny": ["Bash(git push *)", "Read(./.env)"]
    }
  }
  ```

- 每次提示时点「始终允��」会自动写入 `settings.local.json`。

</Accordion>

<Accordion title="如何配置 Hooks 在工具调用前后自动执行脚本？">

在 `.claude/settings.json` 添加：

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {"type": "command", "command": "~/.claude/scripts/guard.sh"}
        ]
      }
    ],
    "SessionStart": [
      {
        "hooks": [
          {"type": "command", "command": "git branch --show-current"}
        ]
      }
    ]
  }
}
```

<Callout type="warn" title="Hooks 调试要点">
  - `matcher` **区分大小写**：是 `Bash`、`Edit`、`Write`、`Read`，不是 `bash`。
  - 多个工具用 `|` 分隔，**字符串而非数组**：`"Edit|Write"`。
  - 修改 `settings.json` 后必须重启 CLI 才生效。
</Callout>

`PreToolUse` 脚本退出码：

- `2` — 阻止本次工具调用
- `0` — 放行

</Accordion>

<Accordion title="自定义 slash command 与 sub-agent 放在哪？">

- **个人级**：`~/.claude/commands/<name>.md`、`~/.claude/agents/<name>.md`
- **项目级**：`.claude/commands/<name>.md`、`.claude/agents/<name>.md`

Markdown 文件头部可用 frontmatter 设置：

```markdown
---
description: 提交当前 git 变更
allowed-tools: Bash, Read
model: sonnet
argument-hint: <提交信息>
---

请按 Conventional Commit 规范提交…
```

</Accordion>
</Accordions>
